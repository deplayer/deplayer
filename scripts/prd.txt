1. Introduction
deplayer is a local-first, browser-based media player designed to allow users to play their music and videos from various sources directly within their browser. It emphasizes user control over data, offline accessibility, and a rich media experience. Users can manage their own media collections and access them from any device, with optional synchronization capabilities.

2. Goals

Unified Media Playback: Provide a single application to play music and video content from diverse sources.

User Data Sovereignty: Ensure user media metadata and collections are stored locally in the browser by default, with no third-party data access.

Cross-Device Accessibility: Enable users to access and play their media on various devices through a Progressive Web App (PWA).

Extensible Media Sourcing: Support multiple media providers, allowing users to connect to local files, torrents, streaming services, and custom servers.

Rich User Experience: Offer modern media player features like notifications, media key integration, lyrics, and visualizations.

Optional Synchronization: Allow users to optionally self-host a sync server to keep their media library and settings consistent across devices.

3. Target Audience

Media Collectors: Individuals with extensive local music and video libraries.

Privacy-Conscious Users: Users who prefer to manage their data locally and avoid third-party cloud services for media playback.

Tech-Savvy Users: Individuals comfortable with self-hosting applications or managing PWA installations.

Multi-Device Users: Users who want a consistent media playback experience across their desktop, laptop, and mobile devices.

Torrent Users: Individuals who consume media via Webtorrent.

Subsonic Users: Users who already manage their media via a Subsonic server.

4. Proposed Solution & Features

4.1. Core Playback Engine
* 4.1.1. Media Playback:
* Support playback of common audio and video formats directly in the browser.
* Standard playback controls: play, pause, stop, next track, previous track, volume control, seek bar.
* 4.1.2. Keyboard Hotkeys:
* space: Play/Pause
* Arrow right or j: Next track
* Arrow left or k: Prev track
* Control + k: Open command bar

4.2. Media Providers & Management
* 4.2.1. Local File System Access:
* Allow users to add media from their local system folders using the File System Access API.
* Provide a fallback mechanism (e.g., standard file input) for browsers not supporting the API.
* 4.2.2. Webtorrent Support:
* Play media directly from magnet links.
* Play media from .torrent files.
* 4.2.3. Subsonic API Integration:
* Connect to and stream media from a user's Subsonic server.
* 4.2.4. youtube-dl-server Integration:
* Allow users to add YouTube videos (and other sites supported by youtube-dl) to their collection by connecting to a self-hosted youtube-dl-server instance.
* 4.2.5. iTunes API (Testing/Samples):
* Connect to the iTunes API to fetch media samples, primarily for testing and demonstration purposes.
* 4.2.6. Multi-Provider Aggregation:
* Display and manage media from all connected providers in a unified library view.

4.3. User Experience & Interface
* 4.3.1. Progressive Web App (PWA):
* Installable on desktop and mobile devices for an app-like experience.
* Offline access to locally stored data and previously cached media (where feasible).
* 4.3.2. Notifications:
* Utilize the Notification Web API to display desktop and mobile notifications (e.g., for track changes, errors).
* 4.3.3. Media Session API Integration:
* Integrate with the OS media controls (e.g., keyboard media keys, lock screen controls on mobile) for playback control.
* Display current track information on system media UIs.
* 4.3.4. Lyrics Display:
* Fetch and display lyrics for the currently playing song using https://api.lyrics.ovh/v1.
* 4.3.5. Audio Visualizations:
* Integrate "butterchurn" for MilkDrop-style visuals.
* Integrate "react-audio-spectrum" for spectrum audio visualization.
* 4.3.6. Multi-Theme Support:
* Allow users to choose from multiple visual themes for the application interface.
* 4.3.7. Command Bar:
* Provide a quick-access command bar (Ctrl+K) for navigating and executing commands within the app.

4.4. Data Management & Synchronization
* 4.4.1. Local-First Data Storage:
* All user data (library, playlists, settings) stored locally in the browser using PGLite (a SQLite Wasm build).
* 4.4.2. Optional Cross-Device Sync:
* Users can optionally set up and connect to a self-hosted sync server (ElectricSQL server backed by PostgreSQL).
* Synchronization of media collection metadata, playlists, and application settings across devices.
* User-configurable sync server URL and enable/disable toggle.

4.5. Sharing
* 4.5.1. Realtime Media Sharing:
* Allow users to share what they are currently playing with others (details on the mechanism TBD, likely peer-to-peer or via a shared link).
* 4.5.2. Download Request for Shared Media:
* Allow recipients of shared media information to request a download (details on the mechanism TBD).

5. Technical Considerations

Frontend: JavaScript/TypeScript, React (implied by react-audio-spectrum), PGLite, HTML5, CSS3.

Key Web APIs: File System Access API, Notification API, Media Session API, WebRTC (for Webtorrent and potentially sharing).

Sync Server: ElectricSQL, PostgreSQL, Docker (for deployment).

Local-First Architecture: Prioritizes local data storage and functionality.

Browser Compatibility: Focus on modern browsers supporting the required Web APIs. Graceful degradation for non-supported features.

6. Design Considerations (User Interface & User Experience)

Responsive Design: The UI must adapt seamlessly to desktop and mobile screen sizes.

Intuitive Navigation: Easy to browse media, manage providers, and access settings.

Clear Visual Hierarchy: Important controls and information should be easily identifiable.

Performance: The application should be responsive, especially during media playback and library browsing.

Accessibility: Adhere to web accessibility standards (WCAG) where possible.

8. Future Considerations / Out of Scope (for this version)

Advanced Playlist Management: Smart playlists, collaborative playlists.

Additional Media Providers: Support for other cloud storage services (e.g., Google Drive, Dropbox) or streaming platforms (if APIs permit).

Transcoding Capabilities: For incompatible media formats (client-side or server-side).

Built-in youtube-dl Functionality: Removing the need for a separate youtube-dl-server.

More Advanced Sharing Options: e.g., temporary access, password protection.

User Accounts (for non-sync server use cases): While local-first, some users might want profiles without full sync.

Detailed Analytics/Statistics on Media Library.
